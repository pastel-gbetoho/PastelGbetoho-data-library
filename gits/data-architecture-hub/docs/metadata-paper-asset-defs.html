Metadata assets

Asset Category Name Description
Business Domain A “Domain” is a combination of one subject area, one business area and optionally one geographical area that constitutes a consistent grouping of data and/or metadata for data governance purposes. Fine-grained domains usually focus on governance of data (e.g. the domain in charge of the data quality of a claim) or highly specific metadata (e.g. one data element or data quality rule that makes sense in a restricted context). Domains defined at macro level usually focus on governance of shared metadata (e.g. one data element or data quality rule that is widely used around the enterprise).
Business Data Privacy Classification Business Data Element shall be classified as: Personal Data - any data relating to an individual who is or can be identified either from the data or from the data in conjunction with other information ; Sensitive Personal Data (also known as “Special categories of personal data”) - data revealing racial or ethnic religion, political opinions, religious or philosophical beliefs, trade-union membership, and the processing of genetic data or biometric data in order to uniquely identify a person, or data concerning health or sexual orientation.
Business Business Criticality Classification Business Data Element shall be classified as "Business Critical Data Element" due to its significant usage, contribution and reliance in achieving entity's business goals (including compliance requirements). Moreover, inconsistencies/deficiencies in data quality shall significantly impact entity’s business goals around financials and/or compliance and/or operations. Some examples of Business Critical Data include Customer ID, Type of Customer, Customer Name, Gender, Data Of Birth, Customer E-mail ID etc.
Business Business Data Object A “(Business) Data Object” is a key idea, person or thing that provides context for the data collected through the enterprise. Data objects aid understanding and are useful for defining objects against which multiple data elements are collected. For example, “contact” may be defined as a data object, which aids in the application of data elements, including contact date, contact delivery mode and contact customer present status. The use of data objects reduces the repetition of large sections of text across multiple data elements.
Business Business Data Element A “(Business) Data Element” is a fundamental, conceptual unit of data. Often, the terms “variable” and “field” are used synonymously to mean a data element (e.g. Individual Contract Status, Claim Event Date and Employee First Name). A data element is comprised of three components: (1) a data object, (2) a property and (3) a value domain. (1) The data object is the idea, person or thing that is of interest and is being described (e.g. the Customer). (2) The property is the characteristic of the data object that is of interest (e.g. the Preferred Language of the Customer). (3) The value domain establishes the permissible values that can be used to represent the data element. A value domain has a data type (e.g. Boolean, decimal and integer) and, optionally, a minimum length, a maximum length and a decimal precision. A value domain can be enumerated (specified through a list of values of a least two individual permissible values) or non-enumerated. For example, the Customer Preferred Language is represented by a list of language codes from the International Organization for Standardization (ISO 639-1:2002). Data elements can be thought of as belonging to one of the following groups: (a) Input: if a data element is of the input type, this indicates that the data is used or stored essentially as it is collected, without major modification. Most data elements are of this type. (b) Derived: one or more data elements which are modified in some way. For example, multiple data elements might be aggregated together for output or logical purposes. See the table ‘Derived Data Elements’ for more information.
Business Business Term Business term are words or phrases used to describe a thing to express a concept in language or branch of business.
Business Business Data Issue A “Business Data Issue” describes a business problem occurring as the result of a data issue and the impact of this problem either qualitative or quantitative. It also describes the nature of the business impact such as financial, regulatory, legal, reputational, operational…
Business Business Data Quality Rule A “(Business) Data Quality Rule” is a Boolean (true/false or passed/failed) expression to which every data record must evaluate true/passed. It is typically the responsibility of (Business) Data (Domain) Owners, as part of their data governance responsibilities, to define data quality rules for domains they own. They should seek input from other data actors (Data Stewards, Internal Controllers, Data Protection Officers and Data Managers) as they define the data quality rules. It should be possible to run data quality rules against data in place, against data being loaded and against data being edited. The definition, design, and implementation of data quality rules are at the heart of a solid data quality management program. It is the data quality rules that will detect, correct, and prevent poor-quality data and elevate data across the enterprise to data that can be trusted as an asset and used for transacting business and for reporting and analytics.
Business Data Processing Activity A “Data Processing Activity” is the collection and manipulation of data elements to produce meaningful information. It represents any operation or set of operations which is performed on data or on sets of data, whether by automated means, such as collection, recording, organization, structuring, storage, adaptation or alteration, retrieval, consultation, use, disclosure by transmission, dissemination or otherwise making available, alignment or combination, restriction, erasure or destruction. In the context of the EU General Data Protection (GDPR), the record of data processing activities is the subset of data processing activities which collect and manipulate personal data.
Business Data Transfer A “Data Transfer” object describes the legal conditions binding the transmission of data in the context of a processing activity to a third party playing the role of data recipient.
Business Data Recipient A “Data Recipient” is a natural or legal person, public authority, agency or another body, external or internal to the organizational entity, to which the data from a data processing activity is disclosed, whether a third party or not. When the data processing activity involves personal data and/or if the personal data is accessed by a recipient located in a third country (international data transfer), additional information may be required.
Technical Application An “Application” is a classification of computer programs designed to perform specific tasks, such as word processing, database management, or graphics (TOGAF Glossary, 2000).
Technical Data Flow A “Data Flow” object describes the data flows between applications.
Technical Database A database is an organized collection of data, generally stored and accessed electronically from a computer system. The database management system (DBMS) is the software that interacts with end users, applications, and the database itself to capture and analyze the data. Source wikipedia
Technical Dataset A data set is a collection of related, discrete items of related data that may be accessed individually or in combination or managed as a whole entity. Source techtarget
Technical Datafield A data field is a place where you can store data. Commonly used to refer to a column in a database or a field in a data entry form or web form. Source wikipedia
Technical Monitored Population A “Monitored Population” is the collection of all members of a master data object that are eligible to be quality-controlled. For reporting purposes, each data quality rule must be associated to a single master data object (e.g. Customers, Employees, Vendors, Suppliers, Products, Locations, contact mechanisms, Profiles, Accounting items, Contracts and Policies) defining the granularity level at which results of data quality assessments are displayed. A monitored population provides the list of all members, identified using their business references, of this master data object (e.g. the list of all Customer Numbers).
Technical Data Confidentiality Classification Labelling Information Assets as Public, Internal, Confidential, Secret. Please refer to Information Security data classification guidelines.
Operational Master Data Object Member A “Master Data Object Member” is an instance of a “Monitored Population” extracted from source system to be quality-controlled (ex. A client identifier extracted from the CRM for the purpose of applying data quality controls to the related client record).
Operational Technical Data Quality Rule A “Technical Data Quality Rule” is the physical executable component (in most cases an SQL SELECT statement) of the data quality rule description. It is the responsibility of IT to design and develop data quality rules that can then be implemented, executed, and monitored within batch or near real-time applications across the enterprise. Output of a data quality rule is the list of all business references (e.g. contract references/numbers, employee references/numbers, vendor references/numbers…) that do not comply with the rule (e.g. the list of all contract references for which the contract start date is not a valid date).
