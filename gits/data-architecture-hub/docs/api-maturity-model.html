1. Managing your APIs Landscape

Basic Intermediate Advanced
1.1. API STRATEGY
No defined API specific strategy in place. APIs according to project needs. APIs are managed as part of the application product strategy. API strategy drives API development based on consumer need. APIs considered as Products that can bring competitive advantage.
1.2. API GOVERNANCE
There is no specific governance around APIs. API governance exists as a standalone IT practice. API governance is a joint IT and Business practice across the full API ifecycle.
1.3. API STANDARDS
Few standards documented and these are not enforced beyond individuals responsibilities. API standards are documented and enforced through an empowered governance body focused on API management. API standards are documented and are automatically enforced and checked at all stages of the API lifecycle e.g.Â workflow, API linter, etc.
1.4. SECURITY POLICIES
Security policies are applied according to the needs of the project and validated through project collaboration with local security teams. No compliance management in place. Baseline security policies are centrally managed and applied locally by the API development teams with security reviews & approval. Specific API policies are created by the project according to a common framework. Manual compliance management in place. Baseline and project specific security policies are centrally managed and applied to relevant APIs through a centralised APIM platform. Automated compliance management in place.
1.5. API CATALOGUE
APIs are built within and managed through individual projects, no shared catalogue. APIs are planned, duplicates are avoided through the use of a shared catalogue. APIs are assets in their own rights and strategically planned. Centralized catalogue is in place and is actively managed to avoid overlap & duplicates and encourage reuse.
1.6. API BUSINESS OWNERSHIP
APIs are considered by the business as a technical artefact for application communications. APIs are considered by business as a technical option to expose services or data to customers and partners. APIs are part of business value proposition to customers and partners, APIs have business owners & roadmaps.
1.7. API DELIVERY APPROACH
API development is managed as part of a project delivery. API development is managed through an Agile delivery approach, delivery team has access to the business. API development is managed through an Agile Biz/Dev/Sec/Ops team, API dedicated roles are clearly defined, understood and assigned.
1.8. API CONSUMER EXPERIENCE
APIs are difficult to discover. Consumers need to contact the API owner and organise access through a process. Consumers are able to find and sign up to APIs through a developer portal with access to documentation and sandbox environment. Consumers have access to self service tools through a singular developer portal and onboarding process. Portal provides IAM, onboarding, sandbox, business & technical documentation. And are part of a developer community.

2. Managing your APIs

Basic Intermediate Advanced
2.1. API SPECIFICATION APPROACH
APIs are designed from the code, not always properly documented. APIs are first designed through project documentation deliverables. API is plugged to underlying IT but does provides consistent mechanisms. APIs are designed first, base on the business value proposition. Specification is available to all actors early in the development process. API layering approach is taken to decouple consumer API from systems of record or underlying system APIs.
2.2. ALIGNMENT TO GUIDELINES
Ad-hoc API for a single set of requirements without any claim to abstraction. API granularity is not considered. API design is specific to its context (a product, application or business domain) and is done with the appropriate granularity. API is based on AXA standards & guidelines. Granularity and abstraction level are based on the existing business domain model.
2.3. API STYLE
APIs are created using any protocol to address specific integration needs. APIs are designed to use web protocols and be RESTful. APIs are designed to fully leverage the power of the RESTful concepts and utilise resources and HTTP verbs to construct services.
2.4. API DOCUMENTATION
Documentation could be partial or unavailable. Not written to OAS standard. Documentation is technically complete. Written to OAS standard. Documentation is complete and versioned, from both a technical and a business perspective, inc. providing value propositions and use cases and is available through an API portal.
2.5. API VERSIONING
API lifecycle is managed opportunistically. There is usually one version per instance. API lifecycle is managed through versioning with SLAs. Backward compatibility is handled. API and consumer lifecycle are governed. Strict API versions are planned, advanced backward compatibility is handled, roadmap is defined, deprecation is managed.
2.6. API DELIVERY (BUILD, DEPLOY, TEST & RELEASE)
Centralized version control, automated build scripts, no management of build artifacts. Automation of builds/tests on CI server. Standard processes for all environments. End-to-end test scenarios are defined and manually executed. Continuous integration and delivery supporting Zero downtime deployments and continue testing of PROD (when possible).
2.7. API RUN (BAU) INC. MONITORING
Simple logging and manual discovery of incidents for technical and security monitoring. Security monitoring of the underlying technology platform is relied upon where relevant. Monitoring & alerting of service usage, performance, logging and manual discovery of incidents. API specific logs gathered by SOC/SIEM for security event detection and incident management. IPC in place and R&R defined. Functional Monitoring. Real-time dashboard and Event Correlation for predictive alerts. Integration of threat intelligence to provide proactive alerting. Business Activity Monitoring (BAM) inc. Business process view. Automatic security monitoring through centralised security monitoring tools, fully integrated with the monitoring of the underlying technology platform.
2.8. API CONTRACT MANAGEMENT
API contracts cover access and use without SLAs. API contracts cover the basic T&Cs of access & usage. Are based off monitoring real world usage & performance data. Consumers are automatically alerted when usage limits are reached and of SLA breaches. API Contracts handle full T&Cs including versioning, retirement etc. Are published and managed through the API Gateway. Relationship with consumers are clearly defined, monitored & controlled.
